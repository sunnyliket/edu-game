## 📋 프로젝트 시작 시 체크리스트

### 매 세션 시작 시 기록할 사항
프로젝트를 시작할 때마다 다음 정보를 기록하세요:

1. **세션 목표**: 이 세션에서 달성해야 할 주요 목표
2. **작업 범위**: 수정할 파일 및 기능 영역
3. **이전 상태**: 프로젝트의 현재 상태 (정상 작동, 에러 등)
4. **예상 결과**: 이 세션이 끝났을 때의 기대 상태

### 변경 이력 기록
각 세션마다 `CHANGELOG.md` 또는 세션 폴더의 `session_log.md`에 다음을 기록:
```markdown
## [Session Date]
- **목표**: 
- **수정 사항**: 
  - 파일1: 변경 내용
  - 파일2: 변경 내용
- **테스트 결과**: 
- **주의사항**:
```

## 🎯 코드 변경 원칙

### 1. 정확한 변경 (Surgical Changes)
- ✅ 사용자의 요청과 정확히 일치하는 변경만 수행
- ✅ 완전한 솔루션 제공 (부분 수정 지양)
- ❌ 관련 없는 코드 수정 금지
- ❌ 예기치 않은 사이드 이펙트 만들지 않기

### 2. 기존 문제 처리
- 요청과 직접 관련된 버그만 수정
- 기존 문제는 고정하지 않음 (별도 작업으로 분류)
- 변경한 코드에서 발생한 버그는 반드시 수정

### 3. 문서화
- 변경과 직접 관련된 문서만 업데이트
- 주석은 필요한 부분만 작성 (설명이 필요한 로직)
- 자명한 코드에는 주석 불필요

### 4. 검증
- 모든 변경 후 기존 동작이 깨지지 않았는지 확인
- 가능한 경우 테스트 실행
- 충돌 여부 점검

## 🛠️ 도구 사용 최적화

### 병렬 처리 (Parallel Tool Calling)
독립적인 작업은 **한 번에 모두 실행**:
```
✅ 옳은 방법:
- 파일 3개 읽기 → 한 응답에서 view 3개 호출
- 파일 2개 수정 → 한 응답에서 edit 2개 호출

❌ 잘못된 방법:
- 파일 읽기 → 응답 대기 → 다음 파일 읽기 (순차 처리)
```

### 파일 읽기
- **큰 파일** (>10KB): `view_range` 사용으로 특정 섹션만 읽기
- **여러 파일**: 모두 병렬로 읽기
- **전체 파일**: `view` 도구 사용 (50KB 제한)

### 검색 및 탐색
우선순위:
1. Code Intelligence Tools (존재하는 경우)
2. LSP 기반 도구
3. `glob` (패턴 매칭)
4. `grep` (내용 검색)
5. PowerShell (최후의 수단)

### PowerShell 사용
- **동기 실행** (`sync`): 빠른 작업, 빌드, 테스트 (initial_wait: 120+초)
- **비동기 실행** (`async`): 인터랙티브 도구, 감시 모드
- **백그라운드 데몬** (`async`, `detach: true`): 서버, 데몬, 지속 실행 필요
- 명령 체이닝: `&&` 사용으로 여러 명령 결합

## 🔄 작업 흐름

### 계획 모드 사용
1. 복잡한 작업은 `plan.md` 작성
2. 할 일 목록을 SQL `todos` 테이블에 저장
3. `exit_plan_mode`로 사용자 승인 요청
4. 사용자 승인 후 구현 시작

### 상태 추적
SQL 테이블 사용:
```sql
-- 할 일 추가
INSERT INTO todos (id, title, description) VALUES 
  ('task-id', 'Task Title', 'Task description');

-- 작업 시작 시
UPDATE todos SET status = 'in_progress' WHERE id = 'task-id';

-- 완료 시
UPDATE todos SET status = 'done' WHERE id = 'task-id';
```

### 의존성 관리
```sql
-- 작업 간 의존성 설정
INSERT INTO todo_deps (todo_id, depends_on) 
VALUES ('task2', 'task1');  -- task1이 완료된 후 task2 시작
```
